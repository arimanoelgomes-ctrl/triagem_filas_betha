## Observação importante sobre esta execução

Esta foi a **primeira execução de validação** do projeto. A postagem via MCP `jira-atendimento__add_comment` foi bloqueada por um bug (o parâmetro `properties` não é repassado à API do Jira — ver `docs/incidente_mcp_add_comment.md`). Como contorno, foi desenvolvido o script `scripts/post_comentarios.js` que chama a API REST diretamente com o payload correto.

### Descobertas técnicas durante a validação

1. **Payload correto da API do Jira da Betha:** capturado via DevTools, exige `properties: [{key: "sd.public.comment", value: {internal: true}}]` para marcar comentário como nota interna.

2. **Bug do MCP:** o `add_comment` do `@betha/jira-mcp` aceita `properties` no schema mas não o repassa para a API. Comentários acabam públicos (situação crítica, encaminhada ao time mantenedor).

3. **Limitação de encoding do Jira:** caracteres Unicode fora do BMP (emojis modernos como 🤖) causam HTTP 500 silencioso. **Solução:** o script `post_comentarios.js` sanitiza emojis antes de enviar; o template em `CLAUDE.md` foi atualizado para não usar emojis.

4. **HTTP 500 falso-negativo:** quando o body contém emojis, o servidor às vezes **grava o comentário no banco** mas retorna HTTP 500. Hoje observamos isso no BTHSC-312959 — após sanitizar e tentar de novo, o script reportou `IGNORADO` porque o comentário já existia da tentativa anterior. **Implicação:** após qualquer erro 500, é seguro re-rodar o script (a idempotência cobre o caso).

5. **Chamados fechados rejeitam comentários:** o servidor retorna HTTP 400 com mensagem confusa (`"você não tem permissão para comentar esta pendência"`). **Solução:** o script verifica o status do issue antes de postar e ignora se estiver em status encerrado. Adicionada nova Regra de Ouro 2 no `CLAUDE.md`.

### Incidente do MCP (registrado em `docs/incidente_mcp_add_comment.md`)

Após capturar o payload real da interface do Jira (via DevTools), o parâmetro correto foi confirmado:

```json
{
    "body": "...",
    "properties": [
        {"key": "sd.public.comment", "value": {"internal": true}}
    ]
}
```

Foi feito um teste de postagem via MCP no BTHSC-318167 com exatamente esses parâmetros (mapeados para `comment` + `explicitUserRequest: true` + `properties: [...]`). **O MCP retornou sucesso, mas o comentário foi gravado como público** — o wrapper não está repassando o array `properties` para a API do Jira.

### Ação tomada

- Comentário **14237696** (BTHSC-318167) com texto "teste interno via MCP" foi **removido imediatamente** via `delete_comment`. Sem prejuízo ao cliente (conteúdo de teste sem informação sensível, intervalo curto).
- Documento técnico criado em `docs/incidente_mcp_add_comment.md` e encaminhado ao time mantenedor do MCP.
- Workaround implementado: script `scripts/post_comentarios.js` chama API REST direta com payload correto.
- 4 comentários de teste no BTHSC-318167 (validação isolada de payload: caso 2, 3, 5, 6) também foram removidos via `delete_comment` ao final do diagnóstico.
- 2 dos 4 comentários da triagem foram efetivamente postados como nota interna em 26/05 (ver tabela abaixo).
